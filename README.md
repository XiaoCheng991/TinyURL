# 企业级目录结构设计（简介架构）
> 采用腾讯云推荐的简洁架构（Clean Architecture），这种分层结构能有效预防代码腐败，确保层级清晰.

```
TinyURL/
├── cmd/                # 程序入口
│   └── server/main.go  # 启动类
├── gateway/            # 接口层：只负责协议解析（Gin Handler），不写业务逻辑
├── logic/              # 逻辑层：核心业务逻辑（如短码生成算法）
├── repo/               # 持久层：MySQL 和 Redis 的具体 CRUD 实现
├── entity/             # 实体层：定义长短链接对应的结构体、错误码
├── config/             # 配置中心：管理数据库连接字符串等
└── go.mod              # 依赖管理文件
```

原因： 这种“分层+模块”的模式让数据流向固定（自上而下），新人接手代码时无需看调用链就能搞清逻辑

## 核心逻辑开发——Base62 算法
> 短域名服务的核心是将 10 进制的 ID 转换为 62 进制的短码（0-9, a-z, A-Z）。

- 编写实体（entity）：在 entity/url.go 中定义基础结构。
- 编写逻辑（logic）：实现一个 Encode(id uint64) string 函数。
- 技术细节：通过余数定理，将数据库自增 ID 映射到 62 位字符表上。
- 进阶优化：在 Go 1.26 中，你可以利用 new(expr) 快速初始化配置结构体，提高代码的声明式阅读感。

## 数据库与缓存集成
- MySQL (Repo层)：在 repo/mysql.go 中编写代码，将 short_code 设为唯一索引 。
- Redis (Repo层)：在 repo/redis.go 中实现缓存。
- 关键策略：查询时“先红（Redis）后黑（MySQL）”，如果 Redis 命中直接返回，大幅降低数据库压力。

## 单元测试（里程碑验证）
> “不写测试的开发不是好程序员” 。Go 语言内置了强大的测试工具。

- 创建测试文件：在同一目录下创建 base62_test.go
- 编写测试函数：函数名必须以 Test 开头，参数为 *testing.T
- 使用 t.Run 进行子测试，分别验证数字 0、大数、异常输入的转换是否符合预期
- 运行测试：执行 go test -v -cover
- 验证标准：测试覆盖率（Cover）应达到 100%，确保核心转换逻辑无死角

## 难点一：如何处理高并发下的重复请求与数据碰撞？
- 挑战：在高并发场景下，同一个URL可能被多次请求生成短码。
- 架构师方案：使用Redis分布式锁确保生成逻辑的原子性。针对ID生成，采用雪花算法避免单机哈希碰撞。
- 为什么选这个：Go天然支持并发，通过sync包或channel可以低成本实现高效的并发控制。

## 难点二：长连接的内存管理与资源回收
- 挑战：成千上万个WebSocket连接如果不及时释放，会导致严重的内存泄漏。
- 架构师方案：利用Go的defer关键字确保资源回收，并结合Go 1.26新引入的Green Tea GC性能优化，显著降低小对象扫描开销。
- 为什么选这个：Go的GMP调度模型能以极小的协程栈（约2KB）支持百万量级的并发连接。
